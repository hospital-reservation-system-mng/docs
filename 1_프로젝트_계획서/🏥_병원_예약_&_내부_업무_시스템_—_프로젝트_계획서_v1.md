## 1. 프로젝트 개요 (Project Overview)

### 1.1 프로젝트 배경

병원의 온라인 접근성 향상 및 환자 편의 증대를 위해 병원 홈페이지를 신규 구축한다.
기존의 전화 중심 예약 방식을 탈피하고, 온라인 예약·관리 통합 플랫폼을 제공하여
환자 경험(Patient Experience) 개선 및 내부 운영 효율화를 목표로 한다.

### 1.2 시스템 구성 요약

본 프로젝트는 아래 5개의 주요 기능 영역으로 구성된다.

| 영역 | 설명 |
| --- | --- |
| 병원 소개 / 과목 / 의사 | Q&A, 시설, 진료과, 의사, 위치·전화번호 안내 |
| 회원가입 & 로그인 | 환자 회원 인증 및 세션 관리 |
| 마이페이지 | 예약 내역 조회, 개인정보 수정 |
| 예약 시스템 | 시간·진료과·의사 선택, 중복 체크, 예약 확정 알림 |
| 관리자 시스템 | 환자 차트, 예약 통계, 게시판, 직원 근무, 문의 내역 관리 |

## 2. 목표 정의 (SMART Goals)

| 기준 | 내용 |
| --- | --- |
| **Specific** | 병원 홈페이지 신규 구축 — 예약, 병원 안내, 관리자 기능 포함 |
| **Measurable** | 온라인 예약 비율 전체 예약의 60% 이상 달성, 관리자 업무 처리 시간 30% 단축 |
| **Achievable** | 기존 병원 운영 데이터 기반 설계, 단계적 기능 개발로 현실 가능한 일정 수립 |
| **Relevant** | 환자 편의성 향상 및 병원 내부 운영 효율화와 직결 |
| **Time-bound** | 착수일로부터 **16주(4개월)** 내 운영 서버 배포 완료 |

## 3. 범위 정의 (Scope Definition)

### ✅ In-Scope (포함 범위)

- 병원 소개 페이지 (Q&A, 시설·장비, 진료과, 의사 프로필, 지도/약도, 전화번호)
- 회원 가입 / 로그인 / 세션 관리
- 마이페이지 (예약 내역, 개인정보 수정)
- 온라인 예약 시스템
    - 시간 단위 예약 슬롯 관리
    - 진료과·의사 선택
    - 중복 예약 체크 (내원 예약 중복 방지)
    - 예약 확정 알림 (이메일 또는 SMS)
    - 입원·수술 예약 연동
- 관리자(Admin) 시스템
    - 환자 차트 (초진/재진/입원 구분)
    - 예약 내역 및 통계 (요일·시간별 현황)
    - 리뷰 등 게시판 관리
    - 직원 근무 현황 (의사, 간호사, 관리인)
    - 문의 내역 관리

### ❌ Out-Scope (제외 범위)

- 전자의무기록(EMR) 시스템 직접 연동
- 의료비 결제(PG) 기능
- 모바일 앱(iOS/Android) 개발
- 다국어(영어 외 언어) 지원
- AI 챗봇 상담 기능

## 7. 형상관리 전략 (Configuration Management)

### 7.1 버전 관리

- **도구:** Git (GitHub 또는 GitLab)
- **브랜치 전략:** Git Flow

```
main          ← 운영 배포 브랜치 (태그 관리)
develop       ← 통합 개발 브랜치
feature/*     ← 기능 단위 개발 브랜치 (예: feature/reservation-api)
hotfix/*      ← 긴급 수정 브랜치
release/*     ← 배포 준비 브랜치
```

### 7.2 코드 리뷰 정책

- PR(Pull Request) 기반 코드 리뷰 필수 (최소 1인 Approve 필요)
- 자동화 린팅(ESLint, Prettier) 및 단위 테스트 통과 시 머지 허용

### 7.3 CI/CD 파이프라인

- Push 시 자동 빌드 및 테스트 실행
- `develop` 브랜치 머지 시 **스테이징 서버** 자동 배포
- `main` 브랜치 머지 시 **운영 서버** 수동 승인 후 배포

### 7.4 환경 구성

| 환경 | 용도 | 접근 권한 |
| --- | --- | --- |
| Local | 개발자 개인 개발 | 개발자 |
| Staging | QA 및 UAT | 개발팀, QA, PO |
| Production | 실 운영 | PM 승인 후 DevOps |

## 8. 리스크 관리 계획 (Risk Management)

### 8.1 리스크 등록부 (Risk Register)

| # | 리스크 항목 | 발생 가능성 | 영향도 | 심각도 | 대응 전략 |
| --- | --- | --- | --- | --- | --- |
| R-01 | 병원 요구사항 변경 (scope creep) | 높음 | 높음 | 🔴 상 | 변경 관리 프로세스 수립, 변경 요청 시 일정·비용 재산정 |
| R-02 | 핵심 인력 이탈 | 중간 | 높음 | 🔴 상 | 업무 문서화 철저, 백업 인력 사전 확보 |
| R-03 | 예약 중복 로직 오류 | 중간 | 높음 | 🔴 상 | 단위 테스트 및 통합 테스트 시나리오 집중 검증 |
| R-04 | 개인정보 보안 취약점 | 낮음 | 매우 높음 | 🔴 상 | 보안 전문가 리뷰, OWASP Top 10 기준 점검 |
| R-05 | 외부 알림 API(SMS/이메일) 장애 | 낮음 | 중간 | 🟡 중 | 재시도 로직 구현, 대체 발송 채널 준비 |
| R-06 | 일정 지연 | 중간 | 중간 | 🟡 중 | 버퍼 기간 확보 (15~16주), MVP 우선 개발 전략 |
| R-07 | 브라우저 호환성 이슈 | 낮음 | 낮음 | 🟢 하 | 크로스 브라우저 테스트 체크리스트 적용 |

### 8.2 리스크 대응 절차

1. **식별(Identify):** 주간 스탠드업에서 새로운 리스크 상시 수집
2. **분석(Analyze):** 발생 가능성 × 영향도 기준으로 심각도 평가
3. **대응(Respond):** 회피 / 완화 / 전가 / 수용 전략 선택 및 실행
4. **모니터링(Monitor):** 매월 리스크 등록부 업데이트 및 이해관계자 보고

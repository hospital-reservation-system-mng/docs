## 1. 프로젝트 개요 (Project Overview)

### 1.1 배경

기존 전화 중심 예약 방식에서 온라인 예약으로 전환하여 운영 효율성을 확보한다.
개발자 중심의 팀 구성을 고려하여 **기능 범위를 최소화하고 안정적 완성**을 최우선 목표로 한다.

### 1.2 목표 정의 (SMART 기준)

| 구분 | 목표 |
| --- | --- |
| **S**pecific | 예약·회원·진료과·의사·관리자 5개 모듈, 관리자 물품 CRUD까지 구현 |
| **M**easurable | 4주 내 MVP 기능 100% 완료 / 주요 예약 흐름 시나리오 테스트 통과 |
| **A**chievable | 책임개발자가 핵심 로직 선점, 개발자는 기능 1개씩 책임 분리 |
| **R**elevant | 전화 예약 → 온라인 전환으로 접수 오류 제거 및 예약 이력 디지털화 |
| **T**ime-bound | 4주차 말 MVP 배포 완료, 추가 기능은 v1.1 이후로 Scope Lock |

## 2. 범위 정의 (Scope Definition)

### 2.1 ✅ In-Scope — 최종 MVP 기능

```
병원 예약 시스템 (MVP v1.0)
├─ 회원
│   ├─ 회원가입
│   ├─ 로그인 (세션 기반)
│   └─ 마이페이지 (예약 조회만)
├─ 진료
│   ├─ 진료과 CRUD
│   └─ 의사 CRUD / 요일 단위 진료 가능 시간 설정
├─ 예약
│   ├─ 진료과 · 의사 · 날짜 · 시간 선택
│   ├─ 고정 30분 슬롯 기반 예약
│   ├─ 동일 시간대 중복 예약 방지
│   └─ 예약 취소
└─ 관리자
    ├─ 예약 조회 & 상태 변경 (완료 처리)
    ├─ 환자(회원) 목록 조회 & 예약 이력 조회
    ├─ 의사 · 간호사 CRUD
    ├─ 물품 등록 · 수량 수정 · 재고 조회
    └─ 단순 통계 (오늘 예약 수 / 전체 예약 수 / 진료과별 count / 재고 부족 수)
```

### 2.2 ❌ Out-Scope — 제거 기능 및 이유

| 제거 기능 | 제거 이유 | 추후 계획 |
| --- | --- | --- |
| JWT 인증 | 구현 리스크, 세션으로 대체 | v1.1 |
| ROLE_DOCTOR / ROLE_NURSE 권한 세분화 | 설계 복잡도 증가 | v1.1 |
| 수술 일정 / 입원 관리 | 도메인 복잡도 높음 | 별도 프로젝트 |
| 진료 상태 복잡 관리 | 고급 상태 머신 불필요 | v1.1 |
| 시간대 동적 생성 | 구현 복잡, 고정 슬롯으로 대체 | v1.1 |
| 근무 스케줄 (근무표) | 일정 관리 로직 복잡 | v1.1 |
| 재고 입출고 로그 | 동시성 문제, 단순 수량 관리로 대체 | v1.1 |
| 재고 자동 알림 | 외부 API 연동 리스크 | v1.1 |
| 복잡한 통계 / 그래프 | 쿼리 난이도 높음 | v1.1 |
| OAuth | 구현 복잡도 | 미정 |
| SMS / 이메일 알림 | 외부 API 리스크 | v1.1 |

### 2.3 기능 난이도 재평가

| 기능 | 난이도 | 비고 |
| --- | --- | --- |
| 세션 기반 인증 | 중 | Spring Security 기본 설정 |
| 예약 핵심 로직 | 중 | 중복 체크, 트랜잭션 처리 |
| 관리자 CRUD | 중 | 반복 구조로 패턴화 가능 |
| 재고 CRUD | 하 | 단순 수량 수정 |
| 통계 (count 집계) | 하 | group by 단순 쿼리 |
| **전체 종합** | **중** | **4주 성공 확률 85~92%** |

## 6. 형상관리 전략 (Configuration Management)

### 6.1 Git 브랜치 전략

```
main          → 배포 가능 안정 버전 (Protected, 직접 커밋 금지)
develop       → 통합 브랜치 (PR 머지 대상)
feature/*     → 기능 단위 개발 브랜치
  feature/auth
  feature/reservation
  feature/admin-patient
  feature/inventory
  feature/statistics
hotfix/*      → 긴급 버그 수정
release/v1.0  → 최종 배포 전 검증 브랜치
```

### 6.2 커밋 컨벤션

```
feat:     새로운 기능 추가
fix:      버그 수정
refactor: 코드 리팩토링
docs:     문서 수정
test:     테스트 코드 추가/수정
chore:    빌드 설정 등 기타

예시)
feat: 예약 중복 방지 로직 구현
fix: ROLE_ADMIN 관리자 접근 오류 수정
feat: 물품 수량 수정 기능 추가
```

### 6.3 코드 리뷰 정책

- `feature/*` → `develop` 머지 시 **책임개발자 리뷰 필수** (PR 기반)
- 리뷰 없는 직접 머지 금지
- 개발자는 PR 설명에 **"구현 내용 / 테스트 방법"** 필수 기재
- AI 보조로 코드 초안 생성 가능하나, 최종 리뷰는 책임개발자가 담당

### 7.1 최고 우선순위 리스크: 개발자 블로킹 (R3)

> 개발자 개발 블로킹은 일정 지연의 가장 큰 원인이다. 아래 3가지 원칙으로 선제 대응한다.
> 
> 1. **기능 1개씩 책임 분리** — 역할 중복 없이 명확한 소유권 부여
> 2. **AI 보조 코드 초안 활용** — 개발자는 초안을 이해하고 수정하는 방식
> 3. **책임개발자 즉각 응답 채널** — 30분 이상 블로킹 시 즉시 에스컬레이션

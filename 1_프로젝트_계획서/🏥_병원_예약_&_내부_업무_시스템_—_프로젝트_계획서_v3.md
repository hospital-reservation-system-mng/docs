# 🏥 병원 예약 & 내부 업무 시스템 — 프로젝트 계획서 v3

> **문서 버전:** v3.0 (시스템 구조 재설계)
**작성일:** 2026년
**개발 기간:** 4주 (1개월)
**팀 구성:** 책임개발자 1명 + 개발자 3명 + AI 보조
**목표 성공 확률:** 90% 이상
> 

---

## 📌 v3.0 재설계 핵심 변경

| 항목 | v2.0 (이전) | v3.0 (현재) |
| --- | --- | --- |
| 외부 사용자 | 회원가입 + 로그인 필요 | **비회원 예약** (이름·연락처만 입력) |
| 내부 사용자 | ROLE_USER / ROLE_ADMIN | **직무별 ROLE** (ADMIN·DOCTOR·NURSE·STAFF) |
| 접수 단계 | 없음 | **예약 → 접수 → 진료 3단계 상태 흐름** |
| 진료 기록 | 없음 | **의사 진료 기록 입력 기능 추가** |
| 회원 테이블 | User 테이블 | **Patient 테이블**로 대체 (비회원) |

### 🎯 왜 이 구조가 더 좋은가?

> **"외부는 단순하게, 내부는 역할에 맞게"**
> 
- 외부 환자는 로그인 없이 예약만 → 진입 장벽 제거, 개발량 감소
- 내부 직원은 직무별 화면 분리 → UI 단순화, Security 구조 명확화
- 예약 → 접수 → 진료의 실제 병원 업무 흐름을 그대로 반영 → 현실적이고 완성도 높은 MVP

## 1. 프로젝트 개요 (Project Overview)

### 1.1 배경

전화 중심 예약 및 수기 관리 방식을 디지털화한다. 외부 환자의 온라인 예약과 내부 직원의 업무 시스템(접수·진료·관리)을 분리하여 현실적인 병원 운영 흐름을 구현한다.

### 1.2 시스템 성격 정의

```
[외부 — 비회원]          [내부 — 직원 로그인]
     예약                  ├─ 접수 (ROLE_STAFF)
  (이름·연락처)            ├─ 진료 (ROLE_DOCTOR)
                           ├─ 간호 업무 (ROLE_NURSE)
                           └─ 전체 관리 (ROLE_ADMIN)
```

### 1.3 목표 정의 (SMART 기준)

| 구분 | 목표 |
| --- | --- |
| **S**pecific | 비회원 예약 + 예약·접수·진료 3단계 상태 흐름 + 직무별 내부 화면 구현 |
| **M**easurable | 4주 내 MVP 100% 완료, 예약→접수→진료 완료 시나리오 E2E 테스트 통과 |
| **A**chievable | 책임개발자가 핵심 구조(Security·상태 흐름) 선점, 개발자는 화면 단위 분담 |
| **R**elevant | 실제 병원 업무 흐름 반영으로 현장 적용 가능한 완성도 확보 |
| **T**ime-bound | 4주차 말 MVP 배포 완료, 추가 기능은 Scope Lock 후 v1.1로 이관 |

## 2. 범위 정의 (Scope Definition)

### 2.1 ✅ In-Scope — 최종 MVP 기능

```
병원 예약 & 내부 업무 시스템 (MVP v1.0)
│
├─ [외부] 예약 (/reservation)
│   ├─ 비회원 예약 폼 (성명·연락처·이메일·진료과·의사·날짜·시간)
│   ├─ 예약번호 발급
│   └─ 중복 예약 방지 (동일 의사·시간대)
│
└─ [내부] 로그인 시스템 (/login)
    │
    ├─ ROLE_ADMIN (관리자)
    │   ├─ 대시보드 (오늘 예약·접수·진료 수 요약)
    │   ├─ 예약 전체 조회 및 관리
    │   ├─ 의사 CRUD
    │   ├─ 직원 CRUD
    │   ├─ 진료과 CRUD
    │   └─ 물품 CRUD (등록·수량 수정·재고 조회)
    │
    ├─ ROLE_DOCTOR (의사)
    │   ├─ 오늘 진료 목록 (/doctor/today)
    │   ├─ 환자 기본 정보 조회
    │   ├─ 진료 내용 입력 (TEXT)
    │   └─ 진료 완료 처리 (RECEIVED → COMPLETED)
    │
    ├─ ROLE_NURSE (간호사)
    │   ├─ 오늘 예약 목록 조회
    │   ├─ 접수 상태 확인
    │   └─ 기본 환자 정보 수정
    │
    └─ ROLE_STAFF (원무/접수)
        ├─ 오늘 예약 목록 (/staff/reception)
        ├─ 접수 처리 (RESERVED → RECEIVED)
        ├─ 환자 추가 정보 입력 (주소·특이사항)
        └─ 전화 예약 등록
```

### 2.2 핵심 상태 흐름 (4단계)

```
RESERVED  →  RECEIVED  →  COMPLETED
 (예약완료)    (접수완료)    (진료완료)

     ↘            ↘            ↘
                CANCELLED (어느 단계에서도 취소 가능)
```

| 상태 | 전환 주체 | 처리 화면 |
| --- | --- | --- |
| RESERVED | 환자 (외부 예약) | /reservation |
| RECEIVED | ROLE_STAFF | /staff/reception |
| COMPLETED | ROLE_DOCTOR | /doctor/today |
| CANCELLED | ROLE_STAFF 또는 ROLE_ADMIN | 각 관리 화면 |

### 2.3 ❌ Out-Scope — 제거 기능

| 제거 기능 | 이유 | 추후 계획 |
| --- | --- | --- |
| 회원 가입 / 로그인 (환자) | 비회원 방식으로 대체 | 불필요 |
| JWT 인증 | 세션 기반으로 단순화 | v1.1 |
| 근무 스케줄 관리 | 일정 관리 로직 복잡 | v1.1 |
| 재고 입출고 로그 | 동시성 문제, 단순 수량 관리로 대체 | v1.1 |
| 재고 자동 알림 | 외부 API 리스크 | v1.1 |
| SMS / 이메일 알림 | 외부 API 리스크 | v1.1 |
| 복잡한 통계 / 그래프 | 단순 count 4종으로 대체 | v1.1 |
| 수술 일정 / 입원 관리 | 도메인 복잡도 | 별도 프로젝트 |
| OAuth | 구현 복잡도 | 미정 |

### 2.4 기능 난이도 평가

| 기능 | 난이도 | 비고 |
| --- | --- | --- |
| 비회원 예약 폼 | 하 | 단순 폼 저장 |
| 중복 예약 방지 | 중 | 트랜잭션 처리 필요 |
| 4 ROLE 권한 분리 | 중 | Spring Security 설정 |
| 예약→접수→진료 상태 흐름 | 중 | 상태 검증 로직 필요 |
| 진료 기록 입력 | 하 | TEXT 저장 CRUD |
| 관리자 CRUD | 하 | 반복 패턴 |
| 단순 통계 4종 | 하 | count group by |
| **전체 종합** | **중** | **4주 성공 확률 85~92%** |

## 3. 데이터 모델 (ERD 요약)

### 3.1 핵심 테이블

```
Patient (비회원 환자)
├─ id (PK)
├─ name       성명
├─ phone      연락처
├─ email      이메일 (선택)
├─ address    접수 시 입력
└─ note       특이사항

Staff (내부 직원)
├─ id (PK)
├─ username   로그인 ID
├─ password   BCrypt 암호화
├─ name       이름
├─ role       ADMIN / DOCTOR / NURSE / STAFF
└─ department_id (FK)

Department (진료과)
├─ id (PK)
└─ name

Doctor (의사 정보)
├─ id (PK)
├─ staff_id   (FK → Staff)
├─ department_id (FK)
└─ available_days  진료 가능 요일

Reservation (예약)
├─ id (PK)
├─ patient_id        (FK)
├─ doctor_id         (FK)
├─ department_id     (FK)
├─ reservation_date
├─ reservation_time
└─ status            RESERVED / RECEIVED / COMPLETED / CANCELLED

TreatmentRecord (진료 기록)
├─ id (PK)
├─ reservation_id    (FK)
├─ doctor_id         (FK)
├─ diagnosis_note    TEXT
└─ created_at

Item (물품)
├─ id (PK)
├─ name
├─ category
├─ quantity
└─ min_quantity  부족 판단 기준
```

### 3.2 핵심 엔티티 관계

```
Patient    1:N  Reservation
Doctor     1:N  Reservation
Reservation 1:1 TreatmentRecord
Department 1:N  Doctor
Staff      1:1  Doctor (의사인 경우)
```

> **책임개발자 W1 내 구조 완성** — Security, ERD, 공통 레이아웃이 완성되어야 개발자 병렬 작업 가능**개발자 기능 1개씩 소유** — 역할 중복 없이 명확한 책임 분리**매주 금요일 통합 미팅** — 브랜치 머지 및 다음 주 계획 확정**Scope Lock 원칙** — 4주 내 신규 기능 추가 절대 금지

## 8. 형상관리 전략 (Configuration Management)

### 8.1 Git 브랜치 전략

```
main              → 배포 가능 안정 버전 (Protected, 직접 커밋 금지)
develop           → 통합 브랜치 (PR 머지 대상)
feature/*         → 기능 단위 개발 브랜치
  feature/auth              인증·권한
  feature/reservation       외부 예약
  feature/reception         접수 처리
  feature/doctor-treatment  진료 기록
  feature/admin             관리자 CRUD
  feature/inventory         물품 관리
hotfix/*          → 긴급 버그 수정
release/v1.0      → 배포 전 최종 검증 브랜치
```

### 8.2 커밋 컨벤션

```
feat:     새로운 기능 추가
fix:      버그 수정
refactor: 코드 리팩토링
docs:     문서 수정
test:     테스트 코드 추가/수정
chore:    빌드 설정 등 기타

예시)
feat: 비회원 예약 폼 구현 및 Patient 저장
feat: 예약 중복 방지 로직 추가 (동일 의사·시간)
fix: ROLE_DOCTOR 접근 권한 누락 수정
feat: 접수 처리 - RESERVED → RECEIVED 상태 변경
```

### 8.3 코드 리뷰 정책

- 모든 `feature/*` → `develop` 머지는 **책임개발자 리뷰 필수** (PR 기반)
- PR 설명에 **"구현 내용 / 테스트 방법 / 스크린샷"** 필수 기재
- AI 보조로 코드 초안 생성 가능 → 최종 리뷰는 책임개발자 담당
